# Vendor Data Flow Analyzer

An AI-powered tool that helps schools and districts investigate edtech vendor data privacy practices — so procurement officers can scrutinize what they're being sold before signing.

---

## The Problem

Schools are pitched edtech tools constantly. Each vendor has a privacy policy, a data processing agreement (DPA), and a subprocessor list — but those documents are dense, inconsistent, and deliberately hard to compare. The vendor's full-time legal team wrote them; the district's procurement officer has an afternoon. This tool closes part of that gap.

---

## What It Does

Enter a vendor or app name (e.g. "ClassDojo", "Seesaw", "Remind"). The AI agent investigates public disclosures and returns a structured report in 1–2 minutes:

### Report Contents

**Risk summary** — A risk-level badge (Elevated / Standard / Low Concern) and four headline metrics:
- Number of direct subprocessors (vendors the app shares data with)
- Estimated downstream vendors (the transitive chain — subprocessors of subprocessors)
- Number of trackers detected via Exodus Privacy static analysis
- Number of discrepancies between declared and detected data practices

**Source documents** — Direct links to the privacy policy, DPA, subprocessor list, App Store listing, and Play Store listing, with last-updated date where available.

**Data flow diagram** — An auto-generated diagram showing which third-party services receive data from the app, color-coded by category (analytics, infrastructure, auth, payments, support).

**Discrepancies flagged** — Side-by-side comparison of what the vendor declares in their documents vs. what static analysis detects. High/medium/low severity. This is the section that changes procurement conversations.

**Subprocessor table** — Every identified vendor with data access, their purpose, data types handled, country of operation, and whether they were declared by the vendor or detected independently.

**Trackers detected** — Results from Exodus Privacy's static Android APK analysis, showing embedded tracking SDKs the app contains.

**Vendor questions** — 5–7 specific, pointed questions generated from the gaps in the vendor's disclosures. These are tied to actual findings, not boilerplate — so the procurement officer walks into the pitch meeting with concrete items the vendor must address.

**Human-in-the-loop checklist** — An honest accounting of what this tool cannot do: dynamic traffic capture, legal interpretation of DPA clauses, evaluation of contractual adequacy under FERPA/COPPA for a specific district's context.

## How It Works

The system is a Claude AI agent with four tools:

| Tool | What it does |
|---|---|
| `search_web` | Brave Search API — finds privacy policy URLs, DPAs, subprocessor lists, news |
| `fetch_page` | Fetches and strips HTML from any URL — reads the actual documents |
| `lookup_exodus` | Queries the Exodus Privacy API for detected trackers in the Android APK |
| `lookup_appstore` | Queries the iTunes Search API for App Store metadata and declared privacy labels |

The agent runs a multi-turn loop: it searches, fetches, reads, and cross-references until it has enough information to produce the report. It makes roughly 8–15 tool calls per analysis. The frontend streams progress in real time so you can watch what it's doing.

The final report is generated by Claude as structured JSON, then rendered in the UI with the diagram, tables, and question list.

### What the agent specifically looks for

1. **Discovery** — Privacy policy URL, DPA URL, subprocessor list URL, App Store and Play Store listings
2. **Document extraction** — Last updated date, data retention policies, deletion timelines, named subprocessors, data types collected, international transfer mechanisms
3. **Static analysis** — Exodus Privacy report: embedded tracker SDKs, permissions requested
4. **Subprocessor chain expansion** — For major declared subprocessors, fetches their own subprocessor lists to estimate the downstream chain
5. **Cross-referencing** — Compares declared subprocessors against detected trackers; flags vendors that appear in one but not the other
6. **Question generation** — Produces vendor-specific questions based on the actual gaps found

### What it cannot do

- **Dynamic traffic analysis**: Observing what data the app actually transmits during a real session requires a physical test device, a logged-in account, and tools like TrackerControl or mitmproxy. The agent can prepare a test plan, but cannot perform the capture.
- **Legal interpretation**: The tool flags DPA clauses that deviate from common templates. Whether a deviation is legally adequate under FERPA/COPPA for your district's specific context requires counsel.
- **Vendor intent**: The tool reports what is and isn't disclosed. It does not adjudicate whether undisclosed practices are intentional or negligent.

## Setup

### Prerequisites

- Node.js 18+
- An [Anthropic API key](https://console.anthropic.com)
- A [Brave Search API key](https://brave.com/search/api/) — free tier (2,000 queries/month) is sufficient

### Installation

```bash
git clone https://github.com/royapakzad/vendor-data-flow.git
cd vendor-data-flow
npm install
cp .env.local.example .env.local
```

Edit `.env.local` and add your keys:

```
ANTHROPIC_API_KEY=sk-ant-...
BRAVE_SEARCH_API_KEY=BSA...
```

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

### Deploying to Vercel

[![Deploy with Vercel](https://vercel.com/button)](https://vercel.com/new/clone?repository-url=https://github.com/royapakzad/vendor-data-flow)

Add `ANTHROPIC_API_KEY` and `BRAVE_SEARCH_API_KEY` as environment variables in your Vercel project settings. The Vercel free tier's 60-second function timeout may be hit on slower analyses — upgrading to Pro (300s timeout) is recommended for production use.

## Limitations and Honest Notes

This tool produces findings from public disclosures and static analysis. A vendor can update their documents at any time; this tool reads them as of the analysis date. The subprocessor chain expansion is one level deep for most vendors — the actual downstream chain may be longer.

The discrepancy flagging is probabilistic. A tracker SDK appearing in Exodus but not in the subprocessor list may have a legitimate explanation (e.g., it's bundled in a dependency, not a direct integration). Use the discrepancy findings as a starting point for questions, not as evidence of wrongdoing.

This tool is most useful as a forcing function: it gives a procurement officer a five-page dossier on a vendor before the pitch meeting, not as a substitute for legal review or dynamic analysis.

---

## Context

Built to address the asymmetry between vendors who have full-time privacy and policy teams and the schools evaluating them — often a single person splitting procurement across dozens of tools. The goal is to change the bargaining dynamic at the table, not to replace legal review.
